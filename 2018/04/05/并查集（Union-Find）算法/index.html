并查集(Union-Find)算法

发表于 2018-04-05 | 分类于 算法 | | 阅读次数:
字数统计: 2,903

 在生活中，连通性其实是个非常常见的问题。比如交通网，互联网，人与人之间的关系网，朋友圈，可能认识的人等等，判断这种连通性，具有很大的价值和意义，这一点，在生活中相信我们自己也深有体会。如下图所示，每一个点可以代表一个网络结点，结点相连即表示网络连通情况，那么，该如何确定节点 p 和节点 q 之间的连通情况呢？

 六度分割理论 则很好的说明了这个世界上人与人之间的连通性。它是说，理论上世界上任意两个不认识的人，只需要通过很少的几个中间人就能建立起联系。当然，这并不是说人与人之间一定要经过六步才能到达，重在表达一种这样的思想：如果 A 想找到 B ，极有可能他们之间本来就是联通的，即 A 通过朋友 1，1 朋友 2，2 通过朋友 3, 3 通过朋友 4， 4 通过朋友 5， 5 和 B 本来就是朋友。

Union Find

Good algorithms can make the difference between being able to solve a practical problem and not being able to address it at all.

 上面的背景中，已经对连通性问题进行了描述。我们需要使用更加精确的数学化的方式来描述它，最终我们需要将它们转化为一个个计算机能够完成的任务。
 我们用整型数据表达一个对象（比如说网络连接中的节点）。我们需要将这些概念抽象化、量化，以便计算机能够处理，而从整型到对象的映射，是非常最简单的一种映射关系。对于一对数据 (p,q) 而言，我们认为它们是相连的，同时，我们这样定义连接：

  • 自反性：p 和 p之间是相连的。
  • 对称性：如果 p 和 q 是相连的，那么，q 和 p 也是相连的。
  • 传递性：如果 p 和 q 是相连的，而 q 和 r 是相连的，则 p 和 r 也相连。

 而我们的问题就是，对于输入的数据 p 和 q ，判断它们是否是连通的。
 更进一步的，计算机的角度 ( API ) 来描述。我们希望这些 API 中能够处理这些事情：我们可以添加相连通的两个节点，来扩充这个连通集；也应该可以判断任意两个整数是否连通；从深度上讲，要进行连通性的判断，还需要首先知道这个整数是否存在于同一个连通集。
 Algorithms 这本书中对这些 API 进行了很好的定义，而且很漂亮。

 这大概就是 Union-Find （合并-查找）名称的来源。通过合并操作来扩大现有连通集，通过查找工作判断连通性。判断连通性有两个非常朴素的方法：

  1. 在一开始合并的时候，我们就给所有连通的元素一个统一的编号来表示它所在的分组，编号一样，在一个分组里，也就是连通的。这样在查询两个元素是否连通就只需要查一下它们的编号是否一致即可。这样做的方式查找速度很多，但是合并速度很慢（因为合并两个集合的时候需要修改其中一个分组中的所有编号）。这被称为 Quick Find 算法。
  2. 另外一种，我们树形结构来表达数据之间的关联，通过一棵树来表达两个数据连接关系，如果两个数据不在同一棵树上，自然就没有连接在一起。这样做，合并（Union）操作很简单，只需要将 q 连接在 p 所在的树上即可，即一次赋值操作。所以，这种方法又叫 Quick Union。

Quick find

 按照上面我们定义好的 API ，现在可以进行完成具体的实现工作了。一种比较简单的思路就是：

  1. 给每一个整型数据（上面我们使用整型来映射一个对象）一个 id ，表示它所在的连通集的编号。
  2. 当两个整型数据具有相同 id 时，则表示它们是连通的， 反之，不连通。
  3. 当进行合并操作的时候，实际上是将两个整型数据所在集合的合并。

 既然代表对象的就是一个整型，而这个整型所对应的 id 也是一个整型，那么使用一个整型数组作为数据结构是一个不错的选择。此时，数组的索引是不可改变的，用以代表这个对象；数组里面的内容，表示 id。

Quick union

 真正非常消耗性能的地方，在于 for 循环的执行。而for 循环的目的，在于帮助我们确定一件事：统一两个集合的编号，将两个“圈子”合并成一个“圈子”。我们可以构建一个关系网，来取代那个关系圈。即，进行 union 操作的时候，不需要通过暴力地将 q 所有所属的数据的集合 id 改成 p 的集合 id 的方式来说明它们是在同一个“圈子”中。我们可以给每个集合指定一个“领头人”，称之为 root 结点，集合中的所有元素都是 root 的结点的子结点。这样，只要两个对象有同一个 root 结点，那么它们自然就是连通的。在进行合并的时候，只需要将其中一个集合的 root 结点挂载到另外一个 root 结点上集合。那么，在查找的时候，实际上我们也是在找它的 root 结点。

在每一层递归上都有三个步骤：

  1. 分解：将原问题分解为若干个规模较小，相对独立，与原问题形式相同的子问题。
  2. 解决：若子问题规模较小且易于解决时，则直接解。否则，递归地解决各子问题。
  3. 合并：将各子问题的解合并为原问题的解。
