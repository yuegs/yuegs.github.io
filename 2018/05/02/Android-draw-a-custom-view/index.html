Android:自定义 View 过程 [译]

发表于 2018-05-02 | 分类于 Android | | 阅读次数:
字数统计: 1,785

 偶然看到一篇关于自定义 View 的绘制流程的短文，觉得还不错，以下是译文。原文地址
 虽然每天我们都面对各种各样的应用，但是它们在设计风格上通常都是类似的。因此，客户通常都希望在布局上体现出自己的差异化。这些有独特风格的设计，通常 Android 系统中内置的 View 是无法完全满足这个需求的，这时，我们就需要自定义这个 View 了。自定义 View 通常会花费更多的时间，但是也更有挑战性、更有趣。

 我最近处理了一个相似的问题：为 ViewPager 创建一个 page indicator 。不像 IOS，Android 并没有提供这样一个 View。我花了很多时间处理这件事，幸运地是最终将它做成一个可以复用的库，为了节省其他开发者的时间，我将它公布在了 github 仓库 里

绘制

 自定义 View 通常会花费更多的时间。所以，当面对一些不好实现的特效或者面临以下问题时，自定义 View 可能是一种比较好的处理方式：

  • 如果你的布局容器中有很多 View，现在你需要将这些 View 进行优化，绘制到一个单独的自定义的 View 中。
  • View 层次太多，同样会带来性能问题。
  • 需要手动去绘制的 View。

 如果你以前没有进行过 View 的自定义，这篇文章是一个机会帮助你定义自己的 View。它将向你展示一个 View 的整体结构，如何去实现特定的需求，如何避免一些错误，以及如何在 View 上运用动画。
 首先，让我们看下 View 的生命周期。

Constructor 构造函数

 每个 View 的生命起点，是从构造函数开始的。这个函数给我们了一个很好的机会进行一些绘制的初始化工作、进行各种运算、设置一些我们需要的默认值等等。
 为了使我们的 View 更容易被创建起来，AttributeSet 是一个很有用的接口。这个接口很容易被实现，也知道我们在它身上花费时间，因为它将帮助你为这个 View 设置一些静态参数。

  • 在创建自定义属性的时候，需要使用一个简单的前缀，避免和其它 View 中相似属性命名冲突。通常，这个前缀可以是 View 名字的缩写，就像 piv_.
  • 如果你使用 Android Studio ，Lint 会建议你在使用完属性之后，调用 recycle() 方法。那是因为它在初始化之后确实不会再被用到，却持有了大量无效数据，需要被回收。

onAttachedToWindow

 当父控件调用了 addView(View) 之后，View 将被附着到 window 上。本阶段，该 View 就可以感知到周围的其它 View 了。如果你的 View 需要和在同一个布局文件中的其它 View 一起协作，那么此时可以通过 findViewById 的方式把它们找出来了。

onMeasure

 在这个阶段，View 需要确定自己的大小。这是一个非常重要的方法，很多情况下，你都需要指定一个大小以适应整体布局。
 当重载这个方法时，你需要 setMeasuredDimension(int width,int height)

 我们可以在布局文件或者程序中设置自定义 View 的大小。为了得到合适的大小，可以这么做：

  1. 设置到你希望定义的 View 的默认大小 desired size（宽度和高度）

  2. 通过 MeasureSpec 获取在布局文件中设置的大小和模式

  3. 通过 MeasureSpec 中的模式，调整 View 的大小。

 注意：让我们来看下 MeasureSpec (译注：MeasureSpec 是父控件提供给子 View 的一个参数，作为设定自身大小参考，只是个参考，要多大，还是 View 自己说了算。)的值：

  • MeasureSpec.EXACTLY:这意味着使用者希望对大小进行硬编码，即写死，就是这个值，不许改变。此时，你需要明确指定它的宽和高。
  • MeasureSpec.AT_MOST:View 可以想多大就多大，只要不超过父控件尺寸即可（译注：对应 wrap_cotent）。
  • MeasureSpec.UNSPECIFIED:父控件对 View 的大小不加任何束缚，想多大就多大。（译注：比如 ScroolView，由于它的特性，对于它的子控件来的高度来说自然是不受限制的）

onLayout

 这个方法，主要是含有子 View 的容器才需要用到，以确定子 View 在容器中的位置。否则，不需覆盖这个方法。

onDraw

 这里就是神奇发生的地方。 Canvas 和 Paint允许你绘制任何你想绘制的东西。
 Canvas 实例通常是作为 onDraw 参数传递过来的，它负责绘制各种形状。而 Paint 对象负责定义形状的颜色。简单来说，Canvas 负责绘制对象，而 Paint 负责给它添加风格。

 在自定义的 View 中，onDraw 方法被调用的次数非常多，以满足绘制和重绘需求。因此，最好不要在这里面创建对象，而是在别的地方创建一次，在这里使用即可。

View Update

 在上面的生命周期图中，你可能看到了有两个方法会导致 View 的重绘：invalidate() 和 requestLayout() (译注：postInvalidate())。

  • invalidate：使用该方法仅仅会重绘 View。比如，当你改变了 View 中的 text、颜色、或者触摸交互，这就意味着只需要调用它的 onDraw 方法即可。
  • requestLayout：正如你所看到的那样，调用这个方法，会导致 View 从 onMeasure 方法重新向下执行。因此，如果你改变了大小，希望它重新测量定位，才需要调用这个方法。（译注：慎用，性能消耗）

Animation

 动画，其实就是自定义 View 一帧一帧变化的过程。比如，你希望一个圆从小到大变化，就需要不断改变它的半径值然后调用 invalidate() 来重绘 View。
 ValueAnimator 将会帮助你完成这个动画，甚至你可以使用插值器。

 注意，在改变了值之后，不要忘记 invalidate()，否则不会重绘，自然也就不会显示出来这种变化。
