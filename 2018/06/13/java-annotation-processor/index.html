Java 注解处理器

发表于 2018-06-13 | 分类于 Java | | 阅读次数:
字数统计: 2,518

 前文中，我们已经知道，从用途和生存期的角度看注解可以分为三类:

  • 编译器指令，用于告诉编译器一些信息，比如 @Override 等，帮助我们纠错。这类注解在生存期上对应于 RetentionPolicy.SOURCE ，在编译的时候，会被编译器丢弃。
  • 编译期指令，运行在编译过程中，可以用来生成类等文件；在生存期上对应于 RetentionPolicy.CLASS ，在编译期存在，但是在虚拟机运行时注解将不被保留。注意，如果不显示指定 Retention 的值，该值将作为默认值。
  • 运行期指令，主要用于反射。在生存期上对应于 RetentionPolicy.RUNTIME ，在编译时和运行时，注解均存在。

 除了编译器指令，后两者都需要一个处理器能够处理它们。今天，就来讲讲这些处理器的自定义过程。

定义注解处理器

 让我们试着创建一个注解处理器。从上面我们已经知道，通常我们会提供的是编译期处理器和运行时处理器。这样，下面将分别介绍。

编译期注解处理器

 这一类注解处理器定义起来有点麻烦，前面已经说了，一般它都会根据注解值输出文件（java源文件或者class文件）。在进行自定义过程之前，我们先来理一下大概思路。
 首先，编译器需要能够认到这个编译处理器，并能够找到它，然后，也是最重要的一点，需要我们自己定义好这个处理过程。
 第一个问题其实比较好解决，为了能够被编译器认到，我们将处理器继承自 AbstractProcessor；为了能够被找到，我们需要把它的位置通知给编译器，这时，我们需要把它打成 jar 包，并在 jar 包里的 META-INF/services 下创建一个 javax.annotation.processing.Processor文件，文件中填写所有自定义注解处理器的全路径,这样，编译器就能够找到、识别并使用它了。
