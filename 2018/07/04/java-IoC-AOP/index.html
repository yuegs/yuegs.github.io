IoC 和 AOP

发表于 2018-07-04 | 分类于 Java |
字数统计: 1,692

 在 ARouter 组件化框架的介绍视频介绍中，提到借鉴了Spring 中 IoC 和 AOP 的思想，为了更好的理解 ARouter,我决定先搞定 IoC 和 AOP 。这两者都是非常优秀的设计思想，以解决大型项目中尤为需要的低耦合、高内聚。

IoC

 IoC (Inversion of Control,控制反转) 是一种用以降低代码之间的耦合的设计原则。所谓反转，即对象 A 需要对象 B 时， B 的创建依赖于外部注入,而非在 A 中创建 。所谓降耦合，即任何单元尽量保持独立，尽量减小相互依赖。这样的好处在于建立良好的扩展性（当项目复杂庞大之后更能体现这一点的优越性）。此时，A 依赖的应是抽象类 B，在使用时注入的是 B 的具体实现类。
 以给电脑（硬件）装系统为例，通俗地讲，将 System1 和 System2 分别装进各自的移动硬盘里，将电脑设置为 U 盘启动，这时，当需要启动 System1 时，就将装有 System1 的硬盘插到电脑上；当需要启动 System2 时，也只是需要将 System2 相应的硬盘插进来即可。事实上，电脑和系统并没有一种强绑定的关系，电脑硬件本身也不生产系统。电脑上运行什么系统，取决于我们往里面注入了什么系统。

AOP

 Aspect-Oriented Programming (AOP)

意为：面向切面编程，通过预编译方式和运行期动态代理实现程序功能的统一维护的一种技术。

 通过上图，解释几个概念先：

  • 切面：如上如所示，切面包含了切入点（pointcut）和通知（advice）。程序本身是按照时间线运行的，而切面就像是在时间线的某个切入点切断，然后添加一层切面进去。
  • 接入点（JointPoint）：就是上面所看到的类中的方法。
  • 切入点（Pointcut）：选择哪个接入点切入，那个切入点就是切面的切入点。上图中，通过通配符的使用，将所有的接入点作为切入点。
  • 目标对象：被一个或者多个切面通告的主体。在 Spring AOP 中，AOP 是通过运行时的动态代理实现的，因此，这个目标对象实际上是个代理对象。

 通告类型：

  • Before advice：在进入接入点之前调用
  • After returning advice：接入点执行完成之后调用
  • After throwing advice：抛出异常时调用
  • After (finally) advice：无论是正常退出还是异常退出之后都会调用
  • Around advice：在方法执行前后和抛出异常时都会执行，相当于以上几种通知。主要用于自定义行为。

小结

 在开发中我们经常讲，程序要高内聚低耦合。IoC 就体现了良好的低耦合，依赖倒置、开放封闭原则（开放扩展，关闭修改）、里氏替换原则（在使用的地方可以由子类代替父类接口）。AOP 体现了良好的高内聚性，单一职责原则（进行职责分配，每个类专职专责）。而这两者，都离不开对接口、抽象的应用。
