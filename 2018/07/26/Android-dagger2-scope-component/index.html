Dagger2---Scope & Component

发表于 2018-07-26 | 分类于 Android | | 阅读次数:
字数统计: 1,889

 最近才开始去了解 Dagger2 ，原本觉着这些注入框架意义不大，Databinding 就足够了。直到后来，看到组件化开发，再去看 IoC，最终还是又回到了依赖注入里。Dagger2 就是利用注解处理器完成动态的代码注入，从而达到低耦合高内聚。对于小项目而言，对于一个新手，确实略显复杂，熟悉了之后可能就轻车熟路了。对于一个复杂项目，就像乐高积木，应该会有更多乐趣。
 回到正题。找了一些不错的资料，原本想写个翻译，在了解过程中发现了一些乐趣，还是自己写点吧。本节不再介绍 Dragger2 概念，直接跳到 Scope 部分。

Scope

 域，在程序的世界里是一个非常普遍的概念，是一个作用边界。比如，命名空间（namespace），作用域链（scope chain）。亦如，车牌号：川A，川B，川C。亦如，国家这么大，皇帝治理不过来，干脆就分成各个省。这样，就构成了一个树形的管理域结构。至于好处嘛，显而易见，便于管理。
 在 Dagger 2 中，也有域的概念。首先，我们看下这样一个需求：在项目中，有些工具对象需要是全局的，它的生命周期存在于整个 App 存在区间。但是，同时也会有一些工具甚至业务处理对象是某个对象所独有的，那么它的生命周期就和该页面息息相关，同生共死。如下图所示：

 Context, RxUtils, HttpUtils, ImageUtils 这些就是存在于整个 App 生命周期内的工具。AController 是存在于 A 类页面内的对象。最上层的各种 Presenter 就是每个页面所独有的。
 我们都知道，Dagger 是一个注入工具。那么，为了处理对象的生命周期问题，就借助注解和注入目标的生命周期来帮助我们管理注入对象的生命周期，实现了 Scope 的概念。

Component

 组件。在此处，它将用于称为 Module 和 Inject 之间的桥梁，作为一个 Injector 。通过 Inject 注解告诉 Dagger 2 这个对象需要被注入；通过 Module 注解提供可以被注入的对象的构造方式；借助 Component 注解完成注入。
 为了清晰的逻辑，降低事物的复杂性，我们同样需要将 Component 分解开来，建立好层次关系。如此一来，就需要解决 Component 之间的依赖问题。这两提供两种方案。

 Scope 和 Component 到此就基本上介绍完了，后续还有待完善。我的出发点，是希望学习它的架构思想，然后才是在此基础上的应用。很多事，都是相通的，看到 Scope 管理，component dependency，有没有想到 WBS；有没有想到单一职责原则；有没有想到树型结构。
