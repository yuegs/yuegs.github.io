Dart 中的事件循环 [译]

发表于 2018-08-30 | 分类于 Flutter | | 阅读次数:
字数统计: 3,355

 原文链接
 异步代码于 Dart 中无处不在。很多库中的函数都会返回一个 Future 对象,当然,你也可以注册对事件(单击、IO完成、timer 过期)的处理器。
 本篇文章，将介绍 Dart 中的事件循环架构，以便人们可以写出更好的异步代码。在本文中，你将学会去调度异步任务，并且可以预测其执行顺序。

基本概念

 如果你写过 UI 代码，你可能很熟悉 事件循环（event loop） 和事件队列（event queue）。它将确保图形操作和事件（比如鼠标点击）能够被及时的处理。

Event loops 和 queues

 event loop 的工作就是从 event queue 中取出一个事件（item），并处理它。然后，重复上述两个步骤，只要 queue 中有实践。

Dart 单线程执行模型

 一旦 Dart 函数执行，它将持续执行，直到退出。换而言之，Dart 函数在执行期间，无法被其他 Dart 代码打断。

注意：代码可以通过创建 isolate 的方式并行运行（Dart web app 不允许创建额外的 isolate，但是可以创建 worker）。Isolate 不能共享内存，他们就像是单独的分离的 app，通过消息进行沟通。除了显式指定代码运行在别的 isolate 或者 worker 中，其他代码都运行在 app 的 main isolate 中。更多信息可以访问Use isolates or workers if necessary

Dart 的事件循环和事件队列

 Dart 应用中，有一个事件循环和 两个队列：event queue 和 microtask queue。

  • event queue: 包含了所有的外部事件：I/O，鼠标点击，绘制，定时器，Dart isolate 中的消息等等。
  • microtask queue：事件处理代码有时需要在当前 event 之后，且在下一个 event 之前做一些任务。

 event queue 包含了来自于 Dart 和系统的事件。当前，microtask queue 中仅仅包含了来自于 Dart 的事件。
 如下图所示，当 main() 退出，event loop 开始工作。首先是执行所有 microtask,它实际上是一个 FIFO 队列。接着，它将取出并处理第一个 event queue 中的事件。接着，开始执行循环：执行所有 microtask，接着执行 event queue 中下一个事件。一旦两个 queue 都空了，也就是说没有事件了，就可能会被宿主（比如浏览器）处理了。

注意：如果 event loop 正在执行 microtask 队列中的事件，那么 event queue 中的事件处理将被停止，这就意味着图像绘制、处理鼠标点击，处理 I/O 等等这些事件将无法执行。

 虽然你可以事先知道 task 执行的顺序，但是，你无法知道 event loop 什么时候从队列中取出任务。Dart 的事件处理系统是基于一个单线程循环模型，而不是基于时间系统。举个栗子，当你创建一个延时任务，事件在一个你指定的时间入队。然而，在它前面的事件没有被处理完，它无法被处理。

安排任务

 当你需要指定一些代码在稍后的时间里执行时，你可以使用 dart:async 库中提供的如下的 API:

  1. Future 类，创建一个事件到 事件队列尾部。
  2. 高层级的 scheduleMicrotask() 方法，添加一个事件到 microtask queue 的尾部。

注意：schduleMicrotask() 现在被命名为 runAsync()

 使用这些 API 的示例将在下面进行。

使用合适的队列（通常是 event queue）

 尽可能的使用 Future，即，使用 event queue。从而减轻 microtask queue 对 event queue 的影响。
 如果一个任务（task），必须在所有 event queue 中事件被处理之前完成，通常你应该立即执行它，如果你不能立即执行它，就需要使用 scheduleMicrotask()，将事件添加到 microtask queue 中。举个栗子，在 web app 中，可以使用 microtask 来避免防止过早的释放 js-interop proxy，或者过早的结束一个 indexedDB transaction 或者事件处理器。

 现在来总结下关于 Future 这些有趣的结论：

  1. 传入进 then() 中方法会在 Future 完成之后立即执行。（这个方法并不会进入 event queue，而仅仅是直接被调用）
  2. 如果 Future 已经执行完毕了，此时再调用 then() ，then() 中的方法会被封装成一个 microtask,进入 microtask queue 中
  3. Future 和 Future.delayed() 并不是立即构造完成，而是先添加这个事件到 event queue 中
  4. Future.value() 的构造出来的任务任务是一个 microtask ，这点和 第2条 很像
  5. Future.sync() 同样也构造出来了一个 microtask，同时，它的函数参数立即执行，这点跟 #2 类似

恰当地使用 isolate 或者 worker

 你是否需要运行一个计算密集型的任务呢？为了保证快速的响应，需要将这类任务放在一个单独的 isolate 或者 worker 中。isolate 可能会运行在一个单独的进程或者线程中，具体情况还要看 Dart 实现。在 1.0 版本中，我们不希望 web app 支持 isolate 或者 worker。但是，然而，你可以使用 dart:html Worker class 添加一个 JavaScript worker 到 Dart 的 web app 中。
 应该使用多少 isolate？对于计算密集型任务，你应该建立尽可能多的 isolate 来使用可用的 CPU 资源。但是，这也将降低 CPU 效率。还有一些，比如说 IO 任务，它不会花费很多的 CPU 资源，但是同样也需要使用单独的 isolate。

总结

 你需要理解 Dart 的 event loop，并且知道该如何安排任务。这里再次总结一下 event loop 中的主要概念：

  • Dart app 中的 event loop 执行的任务来自于两个队列： event queue 和 microtask queue
  • 两个队列接收的事件包括：Dart 事件（future、timer、isolate message 等）和系统事件（用户操作，IO 等）
  • 目前，microtask queue 中主要处理来自 Dart 的事件
  • event loop 会先去执行完 microtask queue 中的所有任务，再去处理 event queue 中的任务
  • 一旦两个队列都空了，app 可以退出了（是否退出取决于宿主，比如浏览器）
  • main() 方法和 所有 microtask queue ，event queue 中的任务均运行在 Dart 的 isolate 中。

 当你在安排任务的时候，需要遵循如下原则：

  • 尽可能的使用 event queue（即，使用 Future 和 Future.delayed()）
  • 使用 Future 的 then() 和 whenComplete() 来处理任务顺序
  • 为了避免 event loop 阻塞，尽量避免使用 microtask queue
  • 为了使得 app 尽可能快的响应，避免在 queue 中运行计算密集型任务
  • 在另外的 isolate 或者 worker 中运行计算密集型任务
