HashMap 概述

我们先来大概看下源码中对 HashMap 的注释来了解下它。
它实现了 Map 接口，同时也允许 key 为 null，value 也为 null。它相当于 Hashtable，除了线程不安全和允许 key 和 value 中 null 的出现。
如果假设，hash 函数能够将元素在 桶（bucket）中 均匀分散开来，它将实现对一些基本操作（get 和 put）实现了常量级别的性能。遍历所花的时间，跟 HashMap 实例的容量以及键值对的数量是成比例的，所以，设置好初始化容量（initial capacity） 和装载因子（load factor）很重要。初始容量不宜过大，装载因子不宜过小，视情况而定。
初始容量 和 装载因子 是影响 HashMap 性能的两个重要因素。所谓容量，即指 桶（bucket） 的数量，初始容量是指当 HashMap 被创建时桶的数量。装载因子会影响容量的大小。当实体(即键值对)的数量超过了装载因子和当前容量的乘积（阈值）,它就会自动成倍的扩容.
通常,默认的装载因子是 0.75,这是综合考虑时间和空间平衡的结果.值太高会增加查询时间.注意,如果很多 key 拥有同样的哈希值(也就是 hashCode() 返回的结果)，同样会降低性能（碰撞）。
同时需要注意，HashMap 中的实现不是线程同步的。在意味着多个线程不能同时访问同一个 HashMap，put 和 resize 这些关键操作都不是线程互斥的。为了保护这些操作，使之可以线程同步，可以使用 Collections.synchronizedMap 对 HashMap 进行包裹。

HashMap 的迭代器遵循 fail-fast 机制。在迭代器创建之后，map 的结构不能方法改变，除非迭代器自己去 remove 一个键值对，否则会抛出 ConcurrentModificationException 异常。因此，面对并发修改，迭代器会马上抛出失败，而不是冒险继续执行迭代，避免一些不确定事件发生。

这是一个 与 操作。其中，n 为 2 的 n 次幂，n-1 得到的二进制位会是 “1111…” 这样的形式。那么，(n-1)&hash 将是取得 hash 的前 k 项二进制位。在初始时，n 值很小，那么，实际上取了 hash 的低 k 位完成下标的计算。
如果不将 前 16 位和 低 16 位 异或操作一下，那么 hash 的高 16 位是完全没有用的，这样将增加碰撞冲突的可能。
从另外一个方面讲，异或操作是计算机十分擅长的，这样，为了减少冲突，就牺牲了一些性能。
