我们就像骑在大象上行进。大象力量巨大、惯性极强，想准时抵达目的地，就必须控制颗粒度细、反馈及时。我们享受着大象所带来的速度、视野和力量，却也无时无刻不在和它的惯性博弈。

回到模型，它给社会带来的变革是颠覆性的。但至少在目前，驯象师始终是我们自己——你可以把路交给大象，但你得知道自己要去哪里。

本文，想继续聊聊 SKILL，尝试驾驭大象的路径与方向。

理解模型

理解模型的特性，有助于更好的设计 SKILL。

模型随机性

我们无法想当然地认为，模型一定会基于现状给出最适合当下的方案。模型的输出，实际上伴随着随机性。基于当前模型已知的信息，下一跳有若干种合理、可选的路径，最终它走向哪一条，似乎都没有错，但不一定能对现状最好。
对于那些对严谨、数字有敏感度的业务，我们需要的更多的是确定性。确定性就意味着可控、稳定。
为了保障这种确定性，为了不随机“抽卡”，我们需要用流程和机制，指引模型往我们希望的、更窄的可行域思考，来对抗模型的随机性。
SKILL 中，定义的流程和机制，帮我们实现了确定性的落地。

人机之间新的平衡点

我们需要在模型和自己之间，找到新的平衡点。有助于我们判断 SKILL 应该精细到什么程度，以及哪些地方需要人来兜底。

  • 注意力是稀缺的：模型本身其实是无状态的，它不会记住关于你的任何事情。之所以你觉得它有记忆力，是因为每次交互（准确来说是 API 调用）时，我们将相关的背景和历史带入其中，构成了可以被关注的上下文。物理限制，导致模型能够处理的上下文是非常有限的，这就导致了空间稀缺性。即使在有限的空间内，过多、甚至关联度不高的信息仍然会导致模型记忆力发散，让本应该被关注点的核心被忽略，这就是密度稀缺性。
  • 决策方法论未必是适合你的：模型本身具备较强的决策能力，但它的决策方法论未必适合你，尤其是当你已经有了一套自己的方法论时。灵活与稳定之间，需要找到一个平衡点。

    You can outsource your thinking, but you can’t outsource your understanding.

  • “好”的标准是私人的：真正的”好”很难量化，众口难调。因此，不妨在普适性的基础上，思考并定义属于你自己的”味道偏好”，而不是你也不知道模型在评估的时候用的是哪一套。
  • 一次不够：一次输出的质量往往是不足的——虽然不至于自相矛盾，但遗漏在所难免。你也许已经发现，每次让模型执行 Review，总能找出新的问题。或许，我们不是要消灭问题，而是接受有限、可控的问题。

SKILL 设计

我们可以通过机制，将模型更加个性化和专业化。决策和标准的私人化靠核心原则编码，随机性靠控制模式收窄可行域，偏差累积靠反馈闭环拦截，可观测性让过程不变成黑箱，注意力稀缺靠上下文管理来保效率。

心智锚点：从目的出发

这里的心智锚点，是我们为什么构建这个 SKILL 的原点。
我们应该清晰地知道，这个 SKILL 扮演着什么角色，职责是什么、范围是什么。

  • 定义清楚这个，全文的所有部分，也将是围绕它们展开。
  • 基于此，当前该做什么，缺乏什么，不该有什么就有了依据和标准。

角色定义，未必是需要写入到 SKILL 中，说到底 SKILL 本就是 Agent 在运行过程中引入的片段，角色也可能在其他地方已经定义。职责和范围一般可以定义在核心原则和行为规范中：做与不做。

核心原则和行为规范：不做与做

核心原则的定义，本质上是在为 SKILL 的运行过程建立一套宪法层。

这层约束，本质上是行为的兜底方案。法律本身就是最低行为底线。

比如，我们可以规定 “所有决策必须具备可追踪的 PRD 支持链条”、“禁止基于推断生成关键行为”、“禁止跳过 Planning 阶段直接执行”。

行为规范，是一个全局规范层。比如 “展示性文案统一使用西班牙语”、“变量命名统一使用英文”。

控制模式：选对结构

SKILL 本质上是在用自然语言编程。顺序、分支、循环——这三种结构，理论上足以表达所有逻辑。

顺序，适用于有明确先后依赖的流程。通常划分为若干 Phase，每个 Phase 下再细分若干 Step，整套流程对应一个领域的最佳实践。

分支，适用于意图驱动的场景。本质是一个领域指令集——用户意图与某条指令匹配，执行对应逻辑。

循环，适用于重复执行直到满足条件的场景。循环的触发，可以是模型本身，也可以是人为介入（需要人机对齐）。循环的终止，可以是满足了 checklist，亦或者是达到最大循环次数的兜底策略。

反馈闭环：不要埋雷

主线上核心环节的输出，一旦和我们的预期偏移，如果不及时制止，预期会在下游被进一步传递、放大。经历的若干下游环节之后，我们再想回溯、修正，将变得极其困难，一旦有遗漏，就非常有可能在后续某个环节上，走向隐秘的错误。 这是“埋雷”。

为了避免这类问题，一个有效的方式是在这个节点上增加模型自身、甚至认为的 Review。我们拥抱变更，但无法拥抱过大范围的变更。

可观测性：让过程看得见

让 SKILL 执行流程可观测性，一方面有利于发现在人为 Review 环节发现质量问题，升级 SKILL；另一方面，也是让模型展开思考过程，进行深度思考。尤其是在方案、计划制定环节，这是非常有必要的。
满足可观测性的过程中，将中间核心节点的输出持久化，将有助于在断点处继续执行。

上下文管理：保持专注

当上下文过于庞大，或许上下文信息中关联性较弱，这些都将稀释模型的注意力。
SKILL 中仍然可以完成部分的上下文管理。

  • 将一个相对独立的子流程，运行在 subagent。subagent 拥有自身独立的上下文，不会污染主流程。当 subagent 获取到结果后，再将处理后的结果回到主流程中。
  • 所有的引用、参考文件，应仅在必要的时候必要的时机再读取。
  • 将一个复杂 SKILL 流程拆分成一个 SKILL 组协作完成，每个 SKILL 输出持久化落盘，其实也是在控制上下文大小。SKILL 组采用接力的形式协作，中间结果落盘，这样，每个中间 SKILL 都可以有一个自己独立的上下文。

SKILL 间

任务结构和协调机制（组织结构与角色职责）
对于一个复杂的任务目标，可能是需要一个 SKILL 集协力完成的。这时，我们需要先得想清楚需要哪些角色，以及他们怎么协作。

想象下我们熟悉的组织方式：每一层都有一个 layer leader，负责协调与决策；同层成员分头推进，不同业务方向；层与层之间，决策上移，任务下放。

每个角色，绑定一个 SKILL。职责边界清晰，原则明确，有自己的最佳实践。角色职责的颗粒度，通常也是一个值得斟酌的平衡点。过大，会导致上下文问题；过小，会增加编排的复杂度、噪声放大。

层内也需要定义好：谁踩油门，谁踩刹车，什么时候踩。这样，SKILL 就被一层一层串了起来。

结构清晰，协作有序。
